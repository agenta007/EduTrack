# EduTrack Backend — Complete Guide

A walkthrough of the backend, in the order you'd want to learn it:
**data model (ERD) → layered architecture → file-by-file logic → JWT auth flow (frontend + backend).**

- **Stack:** Spring Boot 3.5, Java 25, PostgreSQL, Spring Data JPA / Hibernate, Spring Security, JJWT, springdoc-openapi.
- **Module root:** `backend/e-journal/`, package `com.edutrack.e_journal`.
- **Companion docs:** [erd.md](erd.md) (diagram source), `erd-1.png` (rendered), [workflow.md](workflow.md), [`api-help.txt`](../api-help.txt).

---

## 1. The data model (ERD)

The database is the foundation. Tables are **generated by Hibernate** from the JPA
`@Entity` classes (`spring.jpa.hibernate.ddl-auto=update`); `sql/create-tables.sql`
is deprecated. The rendered diagram is `doc/erd-1.png`; the source is `doc/erd.md`.

### Core entities

| Table | Meaning | Key relationships |
|-------|---------|-------------------|
| `roles` | ADMIN / HEADMASTER / TEACHER / STUDENT / PARENT | referenced by `users` |
| `users` | every person (login identity) | `roleid → roles` |
| `schools` | a school | `directorid → users` (headmaster) |
| `classes` | a class within a school | `schoolid → schools` |
| `teachers` | TEACHER extension of a user | PK = `users.id` (`@MapsId`), `schoolid → schools` |
| `students` | STUDENT extension of a user | PK = `users.id`, `classid`, `schoolid`, `parentid → users` |
| `subjects` | a subject (Math, Physics…) | M:N with teachers |
| `teacher_qualifications` | join table teachers ↔ subjects | composite PK |
| `schedules` | a timetable slot (class+subject+teacher+term+day+time) | central link table |
| `grades` | a grade (2–6) | `studentid`, `scheduleid` |
| `absences` | an absence record | `studentid`, `scheduleid` |
| `complaints` | a teacher remark | `studentid`, `scheduleid` |
| `school_events` | events/trips | `schoolid`, `classid`, `created_by → users` |
| `school_profiles` | academic tracks (специалности) | `schoolid` |
| `school_schedule_entries` | bell schedule / special days | `schoolid` |
| `school_term_configs` | per-school term settings | `schoolid` (1:1) |

### Key modelling decisions

- **`teachers`/`students` share their PK with `users`** via `@MapsId` — they're "is-a"
  extensions, not separate identities. A teacher's `userid` *is* the user's `id`.
- **No separate `parents` table** — a parent is just a `users` row referenced by
  `students.parentid`.
- **`schedules` is the hub.** Grades, absences, and complaints all hang off a
  `(student, schedule)` pair — a grade is always tied to a specific lesson slot.

## 2. Layered architecture

Every domain (grades, classes, schools…) is a vertical slice through the same layers.
A request flows **down** through them and a response flows **up**:

```
HTTP request (JSON)
   │
   ▼
@RestController   ── validates input (@Valid), checks role (@PreAuthorize), maps HTTP
   │  calls
   ▼
@Service          ── business rules, role-based scoping, entity <-> DTO mapping
   │  calls
   ▼
@Repository       ── Spring Data JPA; generates SQL from method names
   │  queries
   ▼
@Entity / DB      ── Hibernate-managed rows in PostgreSQL
```

On the way back, a **JPA entity** is converted to a **DTO** (in the service) and
serialized to JSON. The frontend never sees entities directly.

### Why each layer exists

| Layer | Package | Responsibility | Why separate |
|-------|---------|---------------|--------------|
| Controller | `controller/` | HTTP routing, validation, authorization, status codes | Keep web concerns out of business logic |
| Service | `service/` | The actual logic & rules | Reusable, testable, transaction boundary |
| Repository | `repository/` | Data access | No hand-written SQL; swap-able |
| Entity | `entity/` | DB table mapping | Single source of schema truth |
| DTO | `dto/` | API request/response shapes | Decouple API from DB; security; validation |

### Worked example — listing class grades

`GET /api/grades/class/5`:

1. **Controller** `GradeController.getByClass(5)` — `@PreAuthorize("hasAnyRole('ADMIN','HEADMASTER','TEACHER')")` runs first; if the role passes, it calls the service.
2. **Service** `GradeService.getByClass(5)` — calls the repository, then maps each `Grade` entity to a `GradeDto` (flattening student/subject/teacher names).
3. **Repository** `GradeRepository.findAllBySchedule_SchoolClass_Id(5)` — Spring Data turns the method name into a JOIN query.
4. **Entity/DB** — Hibernate returns `Grade` rows.
5. Back up: `List<GradeDto>` → Jackson → JSON array → React.

## 3. File-by-file logic

Package layout under `src/main/java/com/edutrack/edutrack/`:

```
EJournalApplication.java   ← Spring Boot entry point (@SpringBootApplication)
config/                    ← cross-cutting configuration
controller/                ← 14 REST controllers
service/                   ← 13 business-logic services
repository/                ← 14 Spring Data JPA interfaces
entity/                    ← JPA entities + enums
dto/                       ← request/response objects
security/                  ← JWT, filters, user-details
exception/ mapper/ util/   ← shared helpers
```

### `config/`
- **`SecurityConfig.java`** — the security filter chain (stateless, CSRF off, CORS for the React origin), public vs. authenticated URL rules, registers the two custom filters, and defines `PasswordEncoder` (BCrypt) + `AuthenticationManager`. `@EnableMethodSecurity` turns on `@PreAuthorize`.
- **`OpenApiConfig.java`** — Swagger/OpenAPI metadata + the `bearerAuth` JWT scheme (drives `/swagger-ui.html`). Documentation only; no runtime effect.

### `controller/` (HTTP layer)
One controller per domain; each declares `@RequestMapping("/api/...")`, per-method
`@PreAuthorize` role gates, `@Valid` on request bodies, and Swagger annotations.
`AuthController` is the exception — it's public (`/auth/**`) and handles login/refresh/logout.
(Full route + role list: [`api-help.txt`](../api-help.txt).)

### `service/` (business logic)
Where the real work happens: validation beyond bean-constraints (e.g. grade value must
be on the Bulgarian scale), **role-based data scoping** (e.g. `ClassService.getAll`
returns all classes for ADMIN, the school's classes for HEADMASTER/TEACHER), and
**entity↔DTO mapping** via private `toDto(...)` helpers. Services are the
transaction boundary and the unit-testable core.

### `repository/` (data access)
Plain interfaces extending `JpaRepository<Entity, Id>`. Methods are **derived queries** —
Spring Data generates SQL from the method name, e.g.
`findAllBySchedule_SchoolClass_Id(Long)` walks `Grade → Schedule → SchoolClass → id`.
No implementation code is written.

### `entity/` (persistence model)
JPA `@Entity` classes mapping to tables, plus enums (`RoleEnum`, `LectureType`,
`EventType`, `ScheduleEntryType`, `SchoolType`). Lombok generates accessors/constructors:
`@Getter @Setter @NoArgsConstructor @AllArgsConstructor @Builder`
(no-arg ctor is required by JPA; all-args backs `@Builder`).

### `dto/` (API boundary)
- **`*Request`** — inbound bodies, carry validation (`@NotNull`, `@NotBlank`, `@Email`, `@DecimalMin`…), minimal fields, no `id`.
- **`*Dto` / `*Response`** — outbound, flattened/enriched, read-only.
- See [the DTO discussion] — `GradeRequest` (3 fields in) vs `GradeDto` (10 fields out with resolved names).

### `security/` (covered in detail in §4)
`JwtUtils`, `JwtAuthenticationFilter`, `CustomUserDetailsService`,
`LocalhostAdminFilter`, `RequestLoggingFilter`.

### `exception/`, `mapper/`, `util/`
Shared cross-cutting helpers (global exception handling, mapping utilities). `mapper/`
is currently minimal — most mapping lives inline in services.

## 4. JWT authentication flow (frontend + backend)

EduTrack uses a **two-token** scheme:

| Token | Lifetime | Stored where | Purpose |
|-------|----------|--------------|---------|
| **Access token** | 24 h (`app.jwtExpirationInMs=86400000`) | JS memory + `Authorization` header | Authorizes every API call |
| **Refresh token** | 7 days | **HttpOnly cookie** (`refreshToken`, path `/auth`) | Silently mints new access tokens |

Keeping the access token in memory (not localStorage) and the refresh token in an
HttpOnly cookie limits XSS exposure: scripts can't read the refresh token, and a page
reload re-derives the access token from the cookie.

### 4.1 Backend building blocks

- **`JwtUtils`** — signs/verifies tokens with HMAC-SHA from `app.jwtSecret`.
  - `generateAccessToken(user)` → claims: `sub`(email), `id`, `email`, `role`, `name`, `iat`, `exp`.
  - `generateRefreshToken(user)` → subject only, 7-day expiry.
  - `validateToken` / `getEmailFromToken` → verify signature+expiry, read subject.
- **`CustomUserDetailsService`** — `loadUserByUsername(email)` loads the `User` and returns Spring `UserDetails` with authority `ROLE_<role>`. Used by both login (via `AuthenticationManager`/`DaoAuthenticationProvider` + BCrypt) and the JWT filter.
- **`JwtAuthenticationFilter`** — on every request, reads `Authorization: Bearer <token>`, validates it, loads `UserDetails`, and sets the `SecurityContext` so `@PreAuthorize` can evaluate roles.
- **`LocalhostAdminFilter`** — dev-only bypass: tokenless loopback requests get all roles (toggle `app.security.allow-localhost`).
- **`SecurityConfig`** — wires the filters before `UsernamePasswordAuthenticationFilter`, stateless sessions, `/auth/**` public.

### 4.2 Login (`POST /auth/login`)

```
Browser (Login.jsx → AuthProvider.login)
   │  { email, password }
   ▼
AuthController.login
   │  authenticationManager.authenticate(email, password)
   │     └─ DaoAuthenticationProvider → CustomUserDetailsService + BCrypt verify
   │  on success:
   │     accessToken  = JwtUtils.generateAccessToken(user)
   │     refreshToken = JwtUtils.generateRefreshToken(user)
   │     Set-Cookie: refreshToken=...; HttpOnly; Path=/auth; Max-Age=7d
   ▼
200 { "accessToken": "<jwt>" }
   │
   ▼
AuthProvider:  setAccessToken(token); api.defaults.headers.Authorization = Bearer token;
               setUser(jwtDecode(token))   // role/id/name read straight from the JWT
```
Wrong credentials → `401 {"message":"Invalid email or password"}`.

### 4.3 Authenticated request

```
React calls api.get('/api/grades/class/5')
   │  axios request interceptor adds:  Authorization: Bearer <accessToken>   (AuthProvider)
   ▼
SecurityFilterChain:
   LocalhostAdminFilter → (skipped, token present)
   JwtAuthenticationFilter → validate token → load UserDetails → set SecurityContext
   │
   ▼
URL rule (.anyRequest().authenticated())  ✓
@PreAuthorize("hasAnyRole(...)")          ✓ / else 403
   ▼
Controller → Service → Repository → DB → DTO → JSON
```

### 4.4 Silent refresh & session restore

- **On page load** `AuthProvider`'s mount effect calls `POST /auth/refresh` (no body —
  the browser auto-sends the HttpOnly cookie). If the cookie is valid, the backend
  returns a fresh access token and the session is restored without re-login.
- **Every 15 minutes** a timer calls refresh to rotate the access token before it
  would matter. (`axiosInstance` uses `withCredentials: true` so the cookie is sent.)
- If refresh fails (cookie expired/missing) → `logout()` clears state → routes redirect to `/login`.

```
AuthProvider.refreshToken / mount effect
   │  POST /auth/refresh   (Cookie: refreshToken=...)
   ▼
AuthController.refresh → validate refresh JWT → new accessToken
   ▼
200 { accessToken }  → setAccessToken + setUser(jwtDecode)
```

### 4.5 Logout (`POST /auth/logout`)
Backend overwrites the refresh cookie with `Max-Age=0` (deletes it). Frontend clears
`accessToken`/`user`. The access token isn't server-side revoked — it simply expires
(stateless design), so logout relies on dropping it client-side + killing the refresh cookie.

### 4.6 Frontend gatekeeping

- **`context/AuthProvider.jsx`** — owns `accessToken`/`user`/`loading`, the login/logout/refresh logic, the axios interceptor, and session restore.
- **`hooks/useAuth.jsx`** — accessor for the context.
- **`router/ProtectedRoute.jsx`** — waits for `loading`; redirects to `/login` if no user, `/unauthorized` if the user's `role` isn't in the route's allowed `roles`.
- **`router/RoleRedirect.jsx`** — sends `/` to the role's dashboard.

The `user.role` used for these UI checks comes from **decoding the JWT** client-side
(`jwtDecode`) — convenient, but note it's display/UX gating only; **real enforcement is
always the backend `@PreAuthorize`**, because the client can't be trusted.

## 5. End-to-end summary

```
Login → JWT (access in memory + refresh cookie)
      → every API call carries Bearer access token
      → JwtAuthenticationFilter authenticates → @PreAuthorize authorizes
      → Controller → Service (logic + scoping + DTO) → Repository → Postgres
      → DTO → JSON → React state → MUI render
      → access token auto-refreshed from the HttpOnly cookie every 15 min / on reload
```

### Security notes (worth hardening)
- `app.security.allow-localhost=true` — dev bypass; **set `false` in prod**.
- Refresh cookie `Secure=false` — **set `true`** under HTTPS in prod.
- `app.jwtSecret` must be a strong, externalized secret (env var), never committed.
- `mockAdminLogin` in `AuthProvider.jsx` is a dev-only shortcut — remove before release.
