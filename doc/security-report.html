Security Findings

CRITICAL SEC-001 Parent IDOR — Student Grades / Absences / Complaints ▾
Description

Parents authenticated with a valid JWT can query /api/grades/student/{studentId}, /api/absences/student/{studentId}, and /api/complaints/student/{studentId} for any arbitrary student ID. No ownership check confirms the authenticated parent is linked to the requested student. By iterating integer IDs a parent can enumerate all student records across every school in the system.

Affected Files
GradeController.java:53 ⎘ AbsenceController.java:53 ⎘ ComplaintController.java:53 ⎘
Recommendation
In each service's getByStudent() method, check whether the authenticated principal holds the PARENT role and, if so, verify the requested studentId is linked to that parent via the Student entity's parent field. Throw 403 Forbidden if ownership is not confirmed. ADMIN, HEADMASTER, and TEACHER roles may be exempted from the ownership check.
CRITICAL SEC-002 LocalhostAdminFilter Grants Full Admin Access by Default ▾
Description

LocalhostAdminFilter automatically injects all roles (ADMIN, HEADMASTER, TEACHER, STUDENT, PARENT) into the Spring Security context for any unauthenticated HTTP request originating from 127.0.0.1. The flag app.security.allow-localhost=true is set by default in application.properties. In a Docker container, Kubernetes pod, or any CI environment where the application is co-located with other processes, any local service or script can bypass authentication entirely and perform arbitrary admin operations.

Affected Files
LocalhostAdminFilter.java:51–59 ⎘ application.properties:22 ⎘
Recommendation
Set app.security.allow-localhost=false as the production default. Use a Spring profile (@Profile("dev")) to register the filter only in development. Add a startup assertion that fails fast if the flag is true and spring.profiles.active is not dev.
CRITICAL SEC-003 JWT Signing Secret Committed to Source Control ▾
Description

The HMAC-SHA256 signing key app.jwtSecret=ChXOzUTGtMGeRxHkFbhOnbrAjEBhnlttLB2w2biETpTPktQsyBr5LsCcu3bNlGou is committed verbatim in application.properties. Any person with read access to the repository — including contributors, GitHub Actions runners, or a leaked archive — can forge valid JWT tokens for any user including system admins. The secret is also present in git history and cannot be fully purged without a force-push rewrite.

Affected Files
application.properties:17 ⎘
Recommendation
Immediately rotate the current key. Replace the hardcoded value with ${JWT_SECRET} and inject via environment variable or a secrets manager (HashiCorp Vault, AWS Secrets Manager). Add application.properties secret values to .gitignore. Use application-prod.properties (excluded from VCS) for production overrides.
HIGH SEC-004 Profile Picture IDOR — No Authorization on GET Endpoint ▾
Description

GET /api/users/{id}/picture carries no @PreAuthorize annotation and performs no ownership or role check. Any authenticated user (including STUDENT or PARENT) can retrieve the profile picture of any other user by iterating integer IDs. This is a horizontal privilege escalation allowing silent enumeration of all uploaded user photos.

Affected Files
UserController.java:82–87 ⎘ UserService.java:73–87 ⎘
Recommendation
Add @PreAuthorize to allow only the picture owner (principal.id == id) or privileged roles (ADMIN, HEADMASTER). Alternatively, expose a single /api/profile/picture endpoint that always serves the authenticated user's own picture.
MEDIUM SEC-005 File Upload Validated by Content-Type Header Only ▾
Description

Profile picture upload (PUT /api/profile/picture) validates the uploaded file by checking contentType.startsWith("image/"). Content-Type is a client-controlled header that can be trivially spoofed. An attacker can upload an SVG containing embedded <script> tags, an HTML file, or other active content disguised as an image. If subsequently served without Content-Disposition: attachment or X-Content-Type-Options: nosniff, this enables stored XSS.

Affected Files
ProfileService.java:42–56 ⎘
Recommendation
Read the first 16 bytes of the upload and validate magic bytes: PNG (89 50 4E 47), JPEG (FF D8 FF), GIF (47 49 46 38). Reject SVG unconditionally. Enforce a maximum file size (5 MB). When serving images, set X-Content-Type-Options: nosniff and serve from a content-delivery origin separate from the API domain.
MEDIUM SEC-006 CORS Configuration Allows Wildcard Request Headers ▾
Description

config.setAllowedHeaders(List.of("*")) permits any HTTP request header from the allowed origin. Combined with setAllowCredentials(true) — which causes browsers to send HttpOnly cookies cross-origin — this unnecessarily widens the attack surface. Certain CDN and reverse-proxy configurations treat wildcard-header credentialed CORS as a policy violation or behave unexpectedly, potentially caching preflight responses incorrectly.

Affected Files
SecurityConfig.java:77 ⎘
Recommendation
Replace the wildcard with an explicit allowlist: List.of("Authorization", "Content-Type", "X-Requested-With"). Also ensure the production frontend origin is set via an environment variable rather than hardcoded to localhost:5173.
MEDIUM SEC-007 Class & Schedule Access Without Enrollment Verification ▾
Description

GET /api/classes/{id} and schedule endpoints grant access to any authenticated role, including STUDENT and PARENT, without verifying that the student is enrolled in the requested class or the parent has a child there. This leaks cross-school metadata — teacher names, subject lists, timetables — to users with no legitimate need for the information.

Affected Files
ClassController.java:54–59 ⎘
Recommendation
For STUDENT and PARENT roles, resolve the requesting user's enrolled class and compare it against the path parameter. If the IDs do not match, return 403 Forbidden. ADMIN, HEADMASTER, and TEACHER roles may query any class within their school's scope.
LOW SEC-008 Development Mock Login Code in Production Bundle ▾
Description

AuthProvider.jsx contains a mockAdminLogin() function (lines 104–120) and a bypass guard if (accessToken === "fake.jwt.token") return; (line 51). Both are present in the production JavaScript bundle. Discovery via source maps or static bundle analysis exposes hardcoded mock credentials and admin session bypass logic, providing a roadmap for attempted exploitation and failing security audits.

Affected Files
AuthProvider.jsx:51 ⎘ AuthProvider.jsx:104–120 ⎘
Recommendation
Wrap the mock code in if (import.meta.env.DEV) { … } so Vite's tree-shaker eliminates it in production builds. Add an ESLint rule (e.g., no-restricted-syntax matching mockAdminLogin) that fails the CI pipeline if the function is detected outside *.dev.* files.

Passing Security Controls

PASS PASS-01 Passwords Hashed with BCrypt ▾

All passwords are stored using BCryptPasswordEncoder. Plain-text passwords are never persisted or logged. The current password is validated via passwordEncoder.matches() before any password-change operation is accepted.

Evidence
UserService.java:101 ⎘ ProfileService.java:30–40 ⎘
PASS PASS-02 JWT in Memory + Refresh Token in HttpOnly Cookie ▾

Access tokens are held in React component state — not localStorage — protecting against XSS-based token theft. Refresh tokens are issued as HttpOnly, path-scoped (/auth) cookies by the backend, preventing JavaScript access entirely.

Evidence
AuthProvider.jsx:context state ⎘ AuthController.java:115–125 ⎘
PASS PASS-03 Stateless Session — No Server-Side Session State ▾

Spring Security is configured with SessionCreationPolicy.STATELESS. No HTTP sessions are created or maintained. This eliminates session fixation, session hijacking, and CSRF vectors that depend on session cookies.

Evidence
SecurityConfig.java:60 ⎘
PASS PASS-04 Method-Level Security Enabled ▾

@EnableMethodSecurity is active, enabling fine-grained @PreAuthorize annotations on individual controller methods. Most endpoints are guarded with explicit role checks rather than relying solely on URL-pattern matching, reducing the blast radius of a misconfigured security filter chain.

Evidence
SecurityConfig.java:28 ⎘
PASS PASS-05 Input Validation with Jakarta Constraints ▾

All inbound request DTOs are annotated with @NotBlank, @Email, @Size, and @Valid, enforced at the controller layer by Spring's binding result. This prevents null injection, oversized inputs, and malformed email addresses from reaching the service layer.

Evidence
CreateUserRequest.java:all fields ⎘ GradeRequest.java ⎘ AbsenceRequest.java ⎘
PASS PASS-06 No SQL Injection Risk — Parameterized JPA Queries ▾

All database access uses Spring Data JPA derived query methods or JPQL with named/positional parameters. No raw SQL string concatenation or dynamic query construction was found in any repository. The ORM layer fully parameterizes all values before execution.

Evidence
UserRepository.java ⎘ GradeRepository.java ⎘ AbsenceRepository.java ⎘
