# EduTrack — Architecture & Workflow

A full-stack school e-journal (електронен дневник). This document explains every
component, the file that implements it, and how a request flows end-to-end.

- **Backend:** Spring Boot 3.5 (Java 25), PostgreSQL, JWT auth, JPA/Hibernate.
- **Frontend:** React + Vite, Material UI, React Router, axios, i18next.
- **DB schema:** see [erd.md](erd.md). API reference: see [`api-help.txt`](../api-help.txt).

---

## 1. High-level workflow

```
Browser (React SPA, :5173)
   │  fetch  (axios, withCredentials + Bearer JWT)
   ▼
Spring Boot (:8080)
   │  SecurityFilterChain  → LocalhostAdminFilter → JwtAuthenticationFilter → RequestLoggingFilter
   ▼
@RestController  (@PreAuthorize role gate)
   ▼
@Service         (business logic, role-based scoping, validation)
   ▼
@Repository      (Spring Data JPA)
   ▼
PostgreSQL  (ejournal db; tables auto-generated by Hibernate)
```

A response travels back up: entity → **DTO** (via service/mapper) → JSON → React state → MUI render.

## 2. Backend

Root package: `com.edutrack.e_journal`
Location: `backend/e-journal/src/main/java/com/edutrack/edutrack/`
Entry point: `EJournalApplication.java`
Config: `src/main/resources/application.properties` (DB URL, JWT expiry, `app.security.allow-localhost`, log path, port 8080).

### 2.1 Layered structure (per domain)

Each domain follows the same vertical slice: **Controller → Service → Repository → Entity**, with **DTO**s at the boundary.

| Layer | Package | Responsibility |
|-------|---------|----------------|
| Controller | `controller/` | HTTP routes, `@PreAuthorize` role gate, request validation (`@Valid`), Swagger docs |
| Service | `service/` | Business rules, role-based data scoping, entity↔DTO mapping |
| Repository | `repository/` | Spring Data JPA interfaces (derived queries) |
| Entity | `entity/` | JPA `@Entity` classes (DB tables) + enums |
| DTO | `dto/` | Request/response shapes (decouples API from entities) |

### 2.3 Security (`security/` + `config/SecurityConfig.java`)

The filter chain (`SecurityConfig.securityFilterChain`) is **stateless** (no HTTP session), CSRF disabled, CORS allows `http://localhost:5173`. Public routes: `/auth/**`, `/swagger-ui/**`, `/v3/api-docs/**`; everything else requires authentication, with per-method `@PreAuthorize` role checks (`@EnableMethodSecurity`).

| Component | File | Role |
|-----------|------|------|
| Security config | `config/SecurityConfig.java` | Filter chain, CORS, `PasswordEncoder` (BCrypt), `AuthenticationManager` |
| JWT auth filter | `security/JwtAuthenticationFilter.java` | Reads `Authorization: Bearer`, validates token, loads `UserDetails`, sets `SecurityContext` |
| JWT utilities | `security/JwtUtils.java` | Generate/validate access tokens, extract email/claims |
| User details | `security/CustomUserDetailsService.java` | Loads a `User` by email → Spring `UserDetails` with `ROLE_*` authorities |
| Localhost dev bypass | `security/LocalhostAdminFilter.java` | When `allow-localhost=true`, loopback requests **without** a JWT get a virtual admin (all roles). Dev only — set `false` in prod |
| Request logging | `security/RequestLoggingFilter.java` | One log line per request → `logs/requests.log` |

**Auth flow (`AuthController`):**
1. `POST /auth/login` → verifies credentials → returns `{accessToken}` (short-lived JWT) **and** sets an HttpOnly `refreshToken` cookie.
2. `POST /auth/refresh` → reads the refresh cookie → issues a new access token (no body).
3. `POST /auth/logout` → clears the refresh cookie.

### 2.4 Cross-cutting

- `config/OpenApiConfig.java` — Swagger/OpenAPI metadata + bearer scheme (`/swagger-ui.html`).
- `exception/`, `mapper/`, `util/` — shared helpers (exception handling, entity↔DTO mapping, utilities).
- Logging config: `logback-spring.xml` → `logs/e-journal.log` (app + stack traces) and `logs/requests.log`.

## 3. Frontend

Location: `frontend/src/`. Tooling: Vite (`vite.config.js`), env in `.env.development` (`VITE_BACKEND_BASE_URL=http://localhost:8080`).

### 3.1 Bootstrapping & providers

```
main.jsx
 └─ <AuthProvider>          // auth state + token lifecycle
     └─ <StudentProvider>   // selected-student context (parent/headmaster drill-down)
         └─ <ThemeProvider> // MUI theme (theme.jsx)
             └─ <BrowserRouter>
                 └─ <App/> → <AppRouter/>
```

| Concern | File | Role |
|---------|------|------|
| Render root + provider nesting | `main.jsx` | Mounts the app, wires global providers, imports i18n + global CSS |
| App shell | `App.jsx` | Renders `<AppRouter/>` |
| MUI theme | `theme.jsx` | Palette, typography |
| i18n | `i18n/index.js`, `i18n/locales/` | Translations (EN/BG), language persisted in `localStorage` |

### 3.2 Auth & HTTP layer

| Component | File | Role |
|-----------|------|------|
| Axios instance | `api/axiosInstance.js` | `baseURL` from env, `withCredentials: true` (sends refresh cookie) |
| Auth context | `context/AuthProvider.jsx` | Holds `accessToken` + decoded `user`; on mount bootstraps via `/auth/refresh`; attaches `Authorization: Bearer` to every request via an axios interceptor; auto-refreshes the token every 15 min; exposes `login`/`logout`/`refreshToken` |
| Auth hook | `hooks/useAuth.jsx` | `useContext` accessor for `{ user, loading, login, logout, ... }` |
| Selected-student context | `context/StudentContext.jsx` | Tracks which student a parent/headmaster is viewing |
| Sidebar context | `context/SidebarContext.jsx` | Open/collapsed nav state |

> The decoded JWT provides `user.role` (`ADMIN`/`HEADMASTER`/`TEACHER`/`STUDENT`/`PARENT`) and `user.id`, used for both routing guards and UI scoping.

### 3.3 Routing & role gating (`router/`)

| Component | File | Role |
|-----------|------|------|
| Route table | `router/AppRouter.jsx` | All `<Route>`s; each protected route declares allowed `roles` |
| Route guard | `router/ProtectedRoute.jsx` | Waits for `loading`; redirects to `/login` if unauthenticated, `/unauthorized` if role not allowed |
| Landing redirect | `router/RoleRedirect.jsx` | `/` → role dashboard (`/admin`, `/headmaster`, `/teacher`, `/student`, `/parent`) |

**Route families** (path → page, role-gated):
- `/admin/*` → `pages/Admin/*` (manage users, schools, subjects)
- `/headmaster/*` → `pages/Headmaster/*` (school, teachers, students, parents, timetable)
- `/teacher/*` → `pages/Teacher/*`
- `/student/*`, `/parent` → `pages/Student/*`, `pages/Parent/*`
- `/select/...` → `pages/Selection Pages/*` (choose school/class/teacher before a data view)
- `/grades|absences|complaints/{,/school/:id,/class/:id,/student/:id,/me}` → matching `pages/Grades|Absences|Complaints/*` (All-schools → School → Class → Student drill-down)
- `/schedule/:classId` → `pages/Schedule/Schedule.jsx`
- `/statistics/*` → `pages/Statistics/*`

### 3.4 Shared UI components (`components/`)

| File | Role |
|------|------|
| `components/Layout.jsx` | Page shell (app bar + sidebar + content) wrapping each page |
| `components/Sidebar.jsx` | Role-aware navigation menu |
| `components/WelcomeBanner.jsx` | Dashboard greeting |
| `components/UserAvatar.jsx` | Fetches/renders a user's profile picture (`/api/users/{id}/picture`) |

### 3.5 The drill-down pattern (grades / absences / complaints)

These three domains share an identical UX, captured by the comment in `AppRouter.jsx`:

- **ADMIN** → `AllSchools*` (table of all schools) → `School*` → `Class*` → `Student*`
- **HEADMASTER** → starts at their own school's classes
- **TEACHER** → must **select a class** first (`Selection Pages/SelectClassFor*View.jsx`) → `Class*` view
- **STUDENT / PARENT** → straight to the single `Student*` view (`/me` for students)

Example end-to-end (teacher opens class grades):
1. Teacher navigates to `/select/class/grades` → `SelectClassForGradesView.jsx` calls `GET /api/classes`.
2. `ClassController.getAll` → `ClassService.getAll` scopes results to the **teacher's school**.
3. Teacher picks a class → routes to `/grades/class/:classId` → `ClassGrades.jsx`.
4. `ClassGrades` loads `GET /api/grades/class/{id}`, `/api/schedules/class/{id}`, `/api/classes/{id}/students` in parallel and renders a journal table (rows = students, columns = subjects); `POST/DELETE /api/grades` to edit (TEACHER-scoped subjects only).

## 4. Running the system

```bash
# Backend (port 8080) — needs PostgreSQL 'ejournal' db running
cd backend/e-journal && ./mvnw spring-boot:run

# Frontend (port 5173)
cd frontend && yarn dev   # or: ./start-frontend.sh
```
