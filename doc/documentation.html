Project Overview

EduTrack is a full-stack school management system (e-journal) designed for the Bulgarian education system. It supports multiple schools, each with its own headmaster, teachers, students, and parents. Data is organised by class, subject, term, and school year.

JWT Authentication Flow

Login
Client                              Backend
  │                                   │
  │  POST /auth/login                 │
  │  { email, password }              │
  │──────────────────────────────────▶│
  │                                   │  AuthenticationManager.authenticate()
  │                                   │  BCrypt password check
  │                                   │  generateAccessToken()   24 h
  │                                   │  generateRefreshToken()   7 days
  │                                   │  Set-Cookie: refreshToken
  │                                   │    (HttpOnly, path=/auth)
  │◀──────────────────────────────────│
  │  { accessToken: "eyJ…" }          │
Authenticated Request
Client                              Backend
  │                                   │
  │  GET /api/profile                 │
  │  Authorization: Bearer eyJ…       │
  │──────────────────────────────────▶│
  │                                   │  JwtAuthenticationFilter:
  │                                   │    1. Extract Authorization header
  │                                   │    2. validateToken() (sig + expiry)
  │                                   │    3. getEmailFromToken()
  │                                   │    4. CustomUserDetailsService
  │                                   │         .loadUserByUsername(email)
  │                                   │    5. Set SecurityContext
  │                                   │  @PreAuthorize role check
  │◀──────────────────────────────────│
  │  200 { profile data }             │
Token Refresh
Client                              Backend
  │                                   │
  │  POST /auth/refresh               │
  │  Cookie: refreshToken=eyJ…        │
  │──────────────────────────────────▶│
  │                                   │  Read cookie → validate
  │                                   │  Extract email → load user
  │                                   │  generateAccessToken()
  │◀──────────────────────────────────│
  │  { accessToken: "eyJ…" }          │

Access Token Claims

ClaimValue
subUser's email address
idUser's database primary key (Long)
emailUser's email address
roleADMIN HEADMASTER TEACHER STUDENT PARENT
namefirstName + " " + lastName
iatIssued-at (epoch seconds)
expiat + jwtExpirationInMs (default 24 h)

Refresh token contains only sub + iat + exp (7 days), stored as HttpOnly cookie at path /auth.

Frontend Token Handling

  • Access token stored in localStorage, attached to every request via Axios interceptor.
  • On 401 response, interceptor calls POST /auth/refresh using the HttpOnly cookie, updates the stored token, and retries the original request.
  • On logout or refresh failure, localStorage is cleared and the user is redirected to /login.

Security Configuration

Filter Chain Order

1
RequestLoggingFilter
Logs every request — method, path, status, duration, IP — to requests.log.
2
LocalhostAdminFilter (dev only)
When app.security.allow-localhost=true, any request from 127.0.0.1 / ::1 without an Authorization header is automatically authenticated with all five roles. Must be disabled in production.
3
JwtAuthenticationFilter
Validates the Bearer token signature and expiry, then populates SecurityContext.
4
Standard Spring Security filters
Authorization, CORS, CSRF (disabled — stateless API), session management (STATELESS).

Public Endpoints

No authentication required:

  • POST /auth/login, POST /auth/refresh, POST /auth/logout
  • GET /swagger-ui/**, /swagger-ui.html, /v3/api-docs/**

All other paths require a valid JWT.

CORS

Allowed Origin

05

Role System

ADMIN
Global superuser. Full access to all schools and all data.
HEADMASTER
Manages one school. Hires/fires staff, configures terms and schedules.
TEACHER
Records grades, absences, complaints for assigned classes.
STUDENT
Read-only access to own grades, absences, and complaints.
PARENT
Read-only access to their child's grades, absences, and complaints.

Enforcement Levels

  1. Method-level — @PreAuthorize("hasRole('ADMIN') or hasRole('HEADMASTER')") on controller methods.
  2. Service-level — ownership checks (e.g. a HEADMASTER may only modify their own school; a TEACHER may only add grades to classes assigned to them).

Request Lifecycle

A complete example — a teacher submitting a grade:

1
User action
Teacher clicks "Add grade" in ClassGrades.jsx.
2
React sends the request
POST /api/grades with Authorization: Bearer eyJ… and body { studentId: 42, scheduleId: 7, value: 5.5 }
3
RequestLoggingFilter
Begins timing the request and logs the inbound line.
4
LocalhostAdminFilter
Authorization header is present → bypass skipped.
5
JwtAuthenticationFilter
Extracts token → validateToken() (HMAC signature + expiry check) → getEmailFromToken() → loadUserByUsername() loads User from DB as TEACHER → sets SecurityContext.
6
@PreAuthorize check
hasRole('TEACHER') or hasRole('ADMIN') or hasRole('HEADMASTER') → PASS
7
GradeService.addGrade()
Validates schedule, student, and value; saves new Grade entity via GradeRepository.
8
Response
201 Created → { id, studentId, subjectId, value, term, … }
9
RequestLoggingFilter completes
Writes to requests.log: POST /api/grades → 201 (12ms) [127.0.0.1]
10
React updates UI
New grade chip appended to the table without a page reload.

Production Checklist

  • Set app.security.allow-localhost=false
  • Use a random JWT secret of at least 64 characters
  • Update CORS allowed origins in SecurityConfig to the real domain
  • Set spring.jpa.hibernate.ddl-auto=validate (never update in production)
  • Run as a dedicated low-privilege system user (see configure-debian.sh for systemd setup)
  • Serve frontend/dist/ via nginx or similar reverse proxy
